# Claude-SKILL

รวม **Claude Code Skills** สำหรับงานเอกสาร งานสำนักงาน และงานพัฒนาซอฟต์แวร์ของหน่วยงานภาครัฐไทย (พัฒนาและใช้งานจริงที่ทีม DGA-SD)

## จุดประสงค์

- รวบรวม skill ที่ใช้ซ้ำได้กับ [Claude Code](https://claude.com/claude-code) ให้ติดตั้งง่าย ใช้ได้ทุกเครื่อง (macOS / Linux / Windows-WSL)
- เน้นงานที่ทีมทำบ่อย เช่น สรุปรายงานการประชุมจาก transcript อัตโนมัติของ MS Teams ให้เป็นรายงานภาษาไทยพร้อมไฟล์ Word
- ออกแบบให้ **ไม่มีข้อมูลส่วนบุคคลใน repo** — ไฟล์รายชื่อบุคคล (roster) เป็น optional และเก็บเฉพาะในเครื่องผู้ใช้เท่านั้น (ดูหัวข้อ PDPA ด้านล่าง)

## ที่มา

skill ชุดนี้เริ่มจากงานสรุปรายงานการประชุมของ DGA ที่ต้องทำประจำ: transcript ที่ได้จาก MS Teams เป็นการถอดเสียงอัตโนมัติซึ่ง**ชื่อคนและศัพท์เทคนิคเพี้ยนมาก** และการจัดรูปแบบรายงาน (ฟอนต์ราชการ TH SarabunPSK, โครงสร้างวาระ, มติ, action items) กินเวลา จึงทำเป็น skill ให้ Claude Code ทำงานนี้ให้จบในคำสั่งเดียว แล้วขยายเป็น repo กลางสำหรับ skill อื่น ๆ ที่จะตามมา

## การติดตั้ง (ภาพรวม — ใช้กับทุก skill)

1. Clone repo นี้:
   ```bash
   git clone https://github.com/DGA-SD/Claude-SKILL.git
   ```
2. Copy โฟลเดอร์ของ skill ที่ต้องการไปไว้ที่โฟลเดอร์ skills ของ Claude Code ในเครื่องคุณ:
   ```bash
   mkdir -p ~/.claude/skills
   cp -r Claude-SKILL/meeting-summary ~/.claude/skills/
   ```
3. เปิด Claude Code แล้วพิมพ์ `/` — ชื่อ skill จะปรากฏในรายการ

> ติดตั้งเฉพาะบางโปรเจกต์ก็ได้ โดย copy ไปไว้ที่ `<โปรเจกต์>/.claude/skills/` แทน

### 1. meeting-summary

**คืออะไร:** แปลง transcript การประชุมดิบ (ถอดเสียงอัตโนมัติจาก MS Teams) เป็น**รายงานสรุปการประชุมภาษาไทย** และสร้างไฟล์ให้ **2 รูปแบบพร้อมกัน** คือ `.md` และ `.docx` (ฟอนต์ TH SarabunPSK ตัวเนื้อหา 16pt หัวข้อไล่ขนาดอัตโนมัติ ตัวเลขอารบิกทั้งฉบับ) จุดเด่นคือช่วย**แก้ชื่อคน/ศัพท์เทคนิคที่ถอดเสียงเพี้ยน**ให้ถูกต้องตามบริบท และ flag ชื่อที่ไม่แน่ใจให้ผู้ใช้ยืนยันท้ายรายงาน

**มี 2 โหมด:**
- **`normal` (default)** — รายงานสรุปโครงสร้างครบ: หัวเรื่อง / วัน-เวลา / ผู้พูดหลักและบทบาท / ที่มา / สิ่งที่นำเสนอรายวาระ / ช่วงถาม-ตอบ / Action Items / เรื่องอื่น ๆ
- **`have-quote`** — รายงานแบบ**อ้างคำพูดรายผู้พูด**: จัดกลุ่มตามวาระ/ประเด็น แล้วยกคำพูดจริงมาเป็น quote พร้อมชื่อผู้พูดและ timestamp (`> **ชื่อ** (นาที:วินาที): "คำพูด"`) โดยเรียบเรียงคำที่ถอดเสียงเพี้ยนให้ถูกต้องตามบริบท (ไม่ใช่ verbatim 100% — จุดที่ไม่แน่ใจกำกับ `[?]` และมีหมายเหตุนโยบายการแก้คำพูดในตัวรายงาน) — เรียกใช้โดยระบุคำว่า **"have-quote" / "แบบมีคำพูด" / "อ้างคำพูด" / "ยกคำพูด"** ในคำสั่ง

**ข้อกำหนดของเครื่อง:**
- `python3` + แพ็กเกจ `python-docx` (`pip3 install python-docx`)
- ฟอนต์ **TH SarabunPSK** ติดตั้งในเครื่องที่จะ*เปิดอ่าน*ไฟล์ .docx

**ติดตั้ง:**
```bash
cp -r Claude-SKILL/meeting-summary ~/.claude/skills/
pip3 install python-docx
```

**วิธีใช้:**
1. เตรียมไฟล์ transcript เป็น `.txt` หรือ `.md` (ถ้ามีแต่ `.vtt` ให้เปิดไฟล์ transcript `.docx` ที่ดาวน์โหลดจาก MS Teams → copy ข้อความทั้งหมด → วางลงไฟล์ `.txt` — skill ไม่รับ `.vtt` โดยตรงเพราะเปลือง token มาก)
2. ใน Claude Code สั่ง:
   ```
   /meeting-summary สรุปรายงานการประชุมจากไฟล์ ./meeting/transcript-ประชุม-X.txt
   ```
3. ตรวจรายชื่อผู้พูดที่ skill flag ว่าไม่แน่ใจ แล้วยืนยัน/แก้ไขกลับไป

**ตัวอย่างการใช้:**

โหมดปกติ (default):
```
/meeting-summary รบกวนสรุปรายงานการประชุม SC 4-2569 จากไฟล์ transcript นี้ ./meeting/SC-4-2569-meeting-transcript.txt
```
ผลที่ได้: รายงานสรุปครบทุกวาระ พร้อมตารางผู้พูดหลัก ตาราง Action Items และรายการชื่อที่ขอให้ผู้ใช้ช่วยยืนยันท้ายบทสนทนา

โหมดอ้างคำพูด (have-quote):
```
/meeting-summary สรุปการประชุม SC 4-2569 แบบมีคำพูด (have-quote) จากไฟล์ ./meeting/SC-4-2569-meeting-transcript.txt
```
ผลที่ได้: รายงานเรียงตามวาระ แต่ละประเด็นยกคำพูดของผู้พูดมาพร้อม timestamp เช่น `> **ประธาน** (1:27:28): "มติคือเห็นชอบ แต่ด้วยข้อสังเกตจากคณะกรรมการ..."`

**ไฟล์ผลลัพธ์อยู่ที่ไหน:** สร้างใน**โฟลเดอร์เดียวกับไฟล์ transcript ต้นทาง** เสมอ (ไม่ส่งข้อมูลออกนอกเครื่อง):
```
โหมด normal:     สรุปการประชุม-<หัวข้อ>.md / .docx
โหมด have-quote: สรุปการประชุม-<หัวข้อ>-have-quote.md / .docx
```

**Optional — ไฟล์รายชื่อบุคคล (roster) เพื่อให้แก้ชื่อแม่นขึ้น:**
transcript อัตโนมัติมักถอดชื่อคนเพี้ยน skill จะแก้ชื่อได้แม่นยำขึ้นมากถ้าคุณสร้างไฟล์รายชื่อขององค์กรไว้ที่:
```
~/.claude/skills/meeting-summary/references/dga-people.md
```
วิธีสร้าง: copy เทมเพลตจาก `references/dga-people.example.md` (ใน repo นี้ — ชื่อในไฟล์เป็นชื่อสมมติทั้งหมด) แล้วแทนที่ด้วยรายชื่อจริงขององค์กรคุณ ภายหลังมีคนเพิ่ม/ชื่อผิด ก็แก้ที่ไฟล์นี้ไฟล์เดียว
ถ้า**ไม่มี**ไฟล์นี้ skill ยังทำงานได้ตามปกติ โดยอาศัยป้ายชื่อผู้พูดภาษาอังกฤษใน transcript + บริบทการประชุม แล้ว flag ชื่อที่ไม่แน่ใจให้ผู้ใช้ยืนยันแทน

### 2. sak-sai (ซักไซ้)

**คืออะไร:** skill ที่ทำให้ Claude **"ถามให้คมก่อนลงมือ"** — เมื่อสั่งงานใหญ่ (เช่น "ร่างเล่ม มสพร. ให้หน่อย") แต่ข้อมูลสำคัญยังไม่ครบ Claude จะไม่เดาแล้วเขียนไปเรื่อย ๆ แต่จะซักไซ้ด้วยคำถาม 2–4 ข้อที่เปลี่ยนผลลัพธ์ได้จริง (รอบเดียวหรือไม่เกิน 2 รอบ) แล้วสรุปเป็น **requirement brief (สรุปโจทย์)** ให้ผู้ใช้ยืนยันก่อนเริ่มงานจริง จุดที่ยังไม่รู้จะถูกทำเครื่องหมาย `……(รอยืนยัน)` เสมอ ไม่แต่งข้อเท็จจริงเอง

**ที่มา:** ดัดแปลงจาก interview pattern ชื่อ "grill-me" ซึ่งทีม DGA-SD ใช้จริงใน **SD-Agent** (ผู้ช่วย AI สำหรับงานร่างมาตรฐาน มสพร. ตลอดวงจร DD → PRD → FD) — การถาม 1 รอบก่อนเริ่ม ประหยัดกว่าการเขียนเล่มผิดโจทย์ทั้งเล่มมาก

**เหมาะกับ:** งานร่างเล่มมาตรฐาน/เอกสารราชการหลายหน้า — มีชุดคำถามมาตรฐาน 8 ข้อสำหรับเล่ม มสพร. (เรื่อง/ขอบเขต, ประเภทเอกสาร DD/PRD/FD, รหัสเล่ม, เล่มแม่ที่ต้องสอดคล้อง, กลุ่มผู้ใช้, คณะทำงาน, กำหนดเวลา, ชั้นข้อมูล) และแกนคำถามแบบทั่วไปสำหรับงานอื่น

**การใช้:** ไม่ต้องเรียกชื่อ skill — แค่สั่งงานใหญ่ตามปกติ ถ้าข้อมูลขาด Claude จะซักไซ้เอง · ถ้าอยากข้ามให้พูดว่า "ใช้ placeholder ได้เลย" Claude จะลงมือทันที

### 3. safe-coding-workflow

**คืออะไร:** workflow ที่บังคับให้ Claude Code แก้โค้ดแบบ**มีวินัยและตรวจสอบได้** แทนที่จะ "แก้แล้วบอกว่าเสร็จ" — เปลี่ยนทุกงานแก้โค้ดให้เป็นลำดับขั้นที่พิสูจน์ได้: **สำรวจโค้ดก่อนแก้ → วางแผนเป็นขั้นเล็ก ๆ → เช็ค edge case → แก้ให้น้อยที่สุด → ตรวจสอบทั้งระบบ → รายงานพร้อมหลักฐาน** หลักการสำคัญของ skill นี้คือ *"ห้ามอ้างว่าอะไรทำงานได้ ถ้ายังไม่ได้เห็นมันทำงานจริง"* — ทุก claim ในรายงานท้ายงานต้องอ้างอิงคำสั่งที่รันจริงและผลลัพธ์ที่เห็นจริง

**ทำงานเป็น 7 เฟส:**

| เฟส | ทำอะไร |
|---|---|
| 0. ทวนโจทย์ | เขียน goal, เกณฑ์ว่า "เสร็จ" คืออะไร (acceptance criteria), ข้อห้าม, คำถามที่ค้าง — ถ้ายังบอกเกณฑ์เสร็จไม่ได้ = ยังห้ามแตะโค้ด |
| 1. สำรวจก่อนแก้ | อ่านโครงสร้างโปรเจกต์ อ่านไฟล์ที่เกี่ยวจริงทั้งไฟล์ (ไม่แก้จาก grep snippet) หา caller/callee และ**รัน test ก่อนแก้**เพื่อรู้ baseline |
| 2. แตกงาน | แบ่งเป็นขั้นเล็กที่มีจุดประสงค์เดียว ย้อนกลับได้ และมีคำสั่ง verify ประจำขั้น |
| 3. เช็คความเสี่ยง | ไล่ checklist edge case: ค่า null/ค่าว่าง, boundary/off-by-one, timezone, concurrency, ข้อมูลเก่าใน DB, ผลกระทบต่อ API เดิม, error path, security |
| 4. ลงมือแก้ | ตามกฎ 12 ข้อ เช่น minimal diff (ไม่แก้นอก scope), ไม่เปลี่ยน API/contract โดยพลการ, **ห้ามลบ/ปิด test เพื่อให้ผ่าน**, ไม่เพิ่ม dependency โดยไม่มีเหตุผล, ห้าม hardcode secret |
| 5. ตรวจสอบทั้งระบบ | ไล่ verification checklist: test ใหม่ต้อง fail บนโค้ดเก่าก่อน pass บนโค้ดใหม่, รัน test/lint/typecheck ด้วย config จริงของโปรเจกต์, อ่าน `git diff` เองทั้งหมด |
| 6. รายงาน | รายงานตามฟอร์แมตมาตรฐาน: Summary / Changes / **Evidence** / Assumptions / Risks — แยกชัดระหว่าง "ตรวจแล้วเห็นจริง" กับ "คาดว่าน่าจะได้แต่ยังไม่ได้ทดสอบ" และรายงานสถานะ blocked ตรง ๆ ถ้าติด |

**เหมาะกับงานแบบไหน:** bug fix, refactor, เพิ่มฟีเจอร์, **อัปเกรด dependency / patch CVE**, migration, งานในโค้ดที่ไม่คุ้นเคย, งานที่กระทบหลายไฟล์ หรือทุกครั้งที่อยากได้คำอธิบายว่า "แก้อะไร เพราะอะไร พิสูจน์ยังไง"

**ข้อกำหนดของเครื่อง:** ไม่ต้องลงอะไรเพิ่ม — ใช้ shell และเครื่องมืออ่าน/แก้ไฟล์ที่ Claude Code มีอยู่แล้ว (จะได้ผลดีที่สุดเมื่อโปรเจกต์มี test runner, linter และ git)

**ติดตั้ง:**
```bash
cp -r Claude-SKILL/safe-coding-workflow ~/.claude/skills/
```

**วิธีใช้:** ใช้ได้ 2 แบบ

1. **เรียกตรง** — พิมพ์ชื่อ skill นำหน้างานที่สั่ง:
   ```
   /safe-coding-workflow ช่วยแก้ bug: token ที่หมดอายุพอดีเป๊ะ ณ วินาทีที่ request เข้ามา ระบบยังปล่อยผ่าน
   ```
2. **ให้ trigger อัตโนมัติ** — ไม่ต้องพิมพ์ชื่อ skill: Claude Code จะหยิบ skill นี้ขึ้นมาเองเมื่องานเข้าเงื่อนไข (แก้โค้ดหลายไฟล์ / โค้ดไม่คุ้น / งานเสี่ยง) โดยเฉพาะเมื่อสั่งด้วยคำว่า **"ระวังด้วย" / "อย่าทำพัง" / "be careful" / "don't break anything"**

**ตัวอย่างการใช้:**

งาน patch CVE (แบบเดียวกับงานจริงของทีม):
```
/safe-coding-workflow อัปเกรด Next.js เป็น 15.2.6 และ React เป็น 19.0.1 เพื่อ patch CVE-2025-55182
ห้ามกระทบฟังก์ชันเดิม และต้อง build ผ่าน
```
ผลที่ได้: Claude จะรัน build/test เก็บ baseline ก่อนแก้ → อัปเกรดทีละขั้น → รัน `npm run build` + test ซ้ำ → ส่งรายงานท้ายงานที่ระบุไฟล์ที่แก้ (`package.json`, `package-lock.json`), คำสั่งที่ใช้พิสูจน์ และความเสี่ยงที่เหลือ

งานแก้ bug ในโค้ดที่ไม่คุ้น:
```
หน้า export รายงานพังเมื่อข้อมูลมีเครื่องหมาย comma ช่วยแก้หน่อย ระวังอย่าให้กระทบ export แบบอื่น
```

**รูปแบบรายงานท้ายงานที่จะได้ทุกครั้ง:**
```markdown
## Summary      — ทำอะไร สถานะ ✅ เสร็จ / ⚠️ เสร็จแบบมีข้อสังเกต / ❌ ติดขัด
## Changes      — ไฟล์ที่แก้ + เหตุผล (บรรทัดละไฟล์)
## Evidence     — คำสั่งที่รันจริง + ผลลัพธ์ที่เห็น สำหรับทุก claim
## Assumptions  — ข้อสมมติที่ตั้งไว้ตรงไหนที่ข้อมูลไม่ครบ
## Risks        — ข้อจำกัด, edge case ที่ตั้งใจไม่ทำ, ของเดิมที่พังอยู่ก่อนแล้ว
```

> 💡 จุดที่ทีมได้ประโยชน์ที่สุด: ส่วน **Evidence** ทำให้ reviewer ตรวจงานได้โดยไม่ต้องเชื่อคำว่า "เสร็จแล้ว" ลอย ๆ และส่วน **Risks** บอกจุดที่ต้องตามต่อชัดเจน

## หมายเหตุด้าน PDPA / ความปลอดภัยข้อมูล

- **ห้าม commit ไฟล์ `references/dga-people.md` (รายชื่อบุคคลจริง) ขึ้น repo** — เป็นข้อมูลส่วนบุคคลตาม พ.ร.บ. คุ้มครองข้อมูลส่วนบุคคล (PDPA) ไฟล์นี้ต้องอยู่เฉพาะในเครื่องผู้ใช้ (`.gitignore` ของ repo นี้กันไว้แล้ว แต่โปรดระวังเมื่อนำไปใช้ที่อื่น)
- transcript การประชุมมีข้อมูลส่วนบุคคล — skill ถูกออกแบบให้ทำงาน**ในเครื่องเท่านั้น**: ไม่อัปโหลด transcript หรือผลสรุปไปบริการภายนอก และไฟล์ผลลัพธ์อยู่ในโฟลเดอร์เดียวกับ transcript
- skill ปฏิบัติต่อเนื้อหา transcript เป็น "ข้อมูล" เท่านั้น ไม่ใช่ "คำสั่ง" เพื่อป้องกัน prompt injection (OWASP LLM01)
